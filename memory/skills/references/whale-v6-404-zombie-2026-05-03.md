## 根因诊断

**直接原因**：代码中 `DATA_API = "https://gamma-api.polymarket.com"`（第32行）使用了错误的API主机名。Gamma API的 `/trades` 端点已被废弃，正确的Data API端点为 `https://data-api.polymarket.com/trades`。

**端点验证**：
```bash
curl https://gamma-api.polymarket.com/trades?limit=1  # 404 ❌ 已废弃
curl https://gamma-api.polymarket.com/markets?limit=1  # 200 ✅ Gamma API仍可用（用于markets查询）
curl https://data-api.polymarket.com/trades?limit=1    # 200 ✅ 正确端点
```

**根本原因分析**：
1. Whale v6的信号获取依赖于 `gamma-api.polymarket.com/trades` — 这是一个**只读Data API查询**，与CLOB交易无关，本应使用data-api域名
2. Polymarket在2026年4月底前后迁移了API架构：Gamma API的 `/trades` 端点被关闭，仅保留 `/markets` 等端点
3. Bot代码中**没有实现错误熔断**（连续N次错误后进入冷却或切换备用端点）
4. Bot代码中**没有指数退避** — 每2秒固定频率高频轮询
5. Bot的错误循环**没有停止条件** — 永远重试直到进程被外部kill

当cron巡检发现Bot运行但零信号时，按顺序排查：

```bash

ERROR_RATIO=$(tail -100 bot.log | grep -c "404\|500\|WARN\|ERROR")
[ $ERROR_RATIO -gt 90 ] && echo "🔴 API ZOMBIE: ${ERROR_RATIO}% errors"

# 2. 如果错误是404，检查API端点是否有效
ENDPOINT=$(grep -oP 'API GET \K(https?://[^\s:]+)' bot.log | head -1)
curl -s -o /dev/null -w "%{http_code}" "$ENDPOINT"
# 404 → 端点已废弃

# 3. 查找Data API替代
ALTERNATIVE=$(echo "$ENDPOINT" | sed 's/gamma-api\./data-api./')
curl -s -o /dev/null -w "%{http_code}" "$ALTERNATIVE"
# 200 → 日志中的替代有效！

# 4. 检查Gamma API的/markets是否仍存活（确认是部分废弃而非完全下线）
curl -s -o /dev/null -w "%{http_code}" "$(echo "$ENDPOINT" | sed 's|/trades|/markets|')"
# 200 → Gamma API仅/trades废弃
```

## 教训

1. **API源可能随时变更但Bot无法自动感知** — Data API、Gamma API、CLOB API都可能在不通知的情况下变更路径
2. **mtime不能单独作为健康指标** — 文件写入活跃 ≠ Bot在做有用工作
3. **必须引入日志内容级分析** — 最近N行中错误占比 > 95% 应触发告警
4. **零信号的Bot值得怀疑** — Whale v6连续3.5小时0信号，但代码逻辑认为「市场冷清」，不会告警
5. **Polymarket有两个不同的API域名**：
   - `gamma-api.polymarket.com` — 仅用于markets/events查询（不用于trades）
   - `data-api.polymarket.com` — 用于实时成交数据（trades）
   - `clob.polymarket.com` — 用于订单簿和交易执行
   - 三者功能有明确分工，不能混用
