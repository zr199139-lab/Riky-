## 因果链全景

```
ACE费率-1.27%/8h(最高) 
  → select_coins()选中ACE(做空方向正确✅)
  → 开仓353.98个 @$0.17, margin=$19.69
  → ACE价格从$0.17持续跌到$0.15(做空赚钱✅)
  → price_pnl计算: (0.17-0.15)/0.17 * 19.69 * 3 = +$1.10
  → $1.10 > $19.69 * 0.05 = $0.98 → 止盈触发
  → 半仓卖出176.99个, 剩176.99个
  → ❌ margin更新BUG: pos["margin"] = 19.69 * (176.99/353.98) = $9.84
  → 价格继续跌, 新price_pnl基于$9.84算 → 比率虚高 → 再止盈
  → 循环直到仓位<1个
  → 总盈利: 方向正确但被$0.36交易费吃光
```

## 根因分析

### Bug #1: margin未等比缩小（直接原因）

```python
# ❌ 错误：半仓后margin不变
pos["qty"] = remaining  # 更新了
# margin没更新 → price_pnl/margin比率虚高

# ✅ 正确：必须等比缩小
pos["margin"] = pos["margin"] * (remaining / (remaining + filled))
```

即使V2修复加了margin更新行，但**price_pnl的计算也在用旧的margin**——因为3分钟后重新fetch_ticker()算的price_pnl是绝对值，除以新margin后比率变成30%+，远超5%阈值。

### Bug #2: 价格微动就触发（设计缺陷）

半仓后margin=$9.84 → 3分钟后再检查，价格再跌0.1% → price_pnl=$0.08
$0.08 / $9.84 = 0.8% < 5% → 不触发
但系统用了绝对值判断：`price_pnl >= margin * 0.05` → $0.08 < $0.49 → 不触发

**但是**：如果price_pnl因为是**累积的**（从开仓算起的总PnL，不是本次循环的增量），那么$0.08+之前锁利的累积 → 超过阈值 → 触发。

**实际触发原因是**：`price_pnl`每次是重新算总PnL（不是增量），所以半仓后剩余的0.69个虽然值小，但(entry_price - cur_price) * qty 一直累积 → 再跌一点就重新触发。

### Bug #3: 没有止盈后冷却（系统性）

没有「一旦止盈触发，该仓暂停N个检查周期」的冷却逻辑。导致每3分钟检查一次→每次都触发。

## 最终修复方案

**去掉半仓止盈，改为全量平仓。小仓位($20/仓)不需要分段。**

```python
# ✅ 最终版：全量止盈，不分半
if price_pnl >= pos["margin"] * TAKE_PROFIT_PCT:
    to_close.append(sym)  # 直接全量，close_position()
    total_pnl += price_pnl
```

## 对于「为什么重复买同一个亏损币」的回答

ACE本身**不亏损**——做空方向正确（价格从$0.17跌到$0.15）。亏损的是交易费：
- 3次开仓(不同进程) × $0.024 = $0.072
- 15次半仓止盈 × $0.024 = $0.36  
- 合计$0.43交易费

**用户看到的「重复买同一个亏损币」其实是：**
1. 黑名单没生效（state被清空→启动后黑名单丢失）
2. ACE费率始终最高(-1.27%/8h) → 每轮结算都选中ACE
3. 价格在跌→做空正确→但止盈循环切光仓位→4h后再开→再循环

## 教训清单

1. **半仓止盈不适合小仓**：<$50/仓的仓位一次全平更优，多次交易费占比太高
2. **margin同步是半仓操作的前提**：半仓后不更新margin等于没半仓
3. **止盈必须有冷却**：触发后跳过N个检查周期，防止价格微动连续触发
4. **state数据一致性**：对state的任何改动都要同步到磁盘+下次读取时验证
5. **启动时强制fetch_positions**：不信任任何本地缓存
